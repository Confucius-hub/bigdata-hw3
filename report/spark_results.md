# ДЗ 3, часть 2 — Результаты Spark

## Конфигурация окружения

- MacBook Air, Apple Silicon (ARM), 8 ГБ RAM
- Docker Desktop с лимитом ~6 ГБ RAM
- Apache Spark 3.5.3 в Docker (образ `apache/spark:3.5.3`)
- Spark Master + 1 Spark Worker
- Worker конфигурация: 4 ГБ памяти, 4 ядра

## Датасет

- Формат: CSV
- Размер: **50.06 GiB** (53.7 ГБ)
- Строк: **789 000 000**
- Колонки: id, user_id, event_type, product_id, category, price, country, event_time
- Генерация: `scripts/generate_dataset.py --target-gb 50`

## Запрос

PySpark job (`jobs/spark_job.py`) выполняет две агрегации:

1. **По стране (`country`):** count events, count distinct users, sum revenue, avg price.
2. **По категории (`category`):** count events, count distinct users, sum revenue.

Обе агрегации требуют **полного прохода** по всему датасету.

Результаты пишутся в Parquet — это материализует вычисления и обеспечивает корректный замер времени.

## Замеры времени

| Режим | Wall-clock | В секундах |
|---|---|---|
| `--master local[*]` | ~3 ч 59 мин | **~14 340 s** |
| `--master spark://spark-master:7077` | ~3 ч 30 мин | **~12 600 s** |

Точные значения получены по timestamp создания выходных Parquet-файлов в `results/local/` и `results/cluster/`. Они отражают полное время от начала до завершения обеих агрегаций.

## Анализ

На эмуляции кластера в Docker на одной физической машине с 8 ГБ RAM **разница между local и cluster минимальна (~10%)**. Причины:

1. Master и Worker делят те же CPU и RAM хоста.
2. На таком объёме (50 ГБ CSV против 4 ГБ доступной памяти у executor) Spark вынужден активно сбрасывать промежуточные данные на диск (`spilling sort data to disk`).
3. CSV-формат сам по себе очень медленный — каждая строка требует парсинга.
4. В кластерном режиме добавляются накладные расходы на сериализацию/десериализацию между Master и Worker по сети контейнеров.

В реальном кластере на разном железе кластерный режим практически всегда быстрее локального — там Worker'ы работают параллельно на разных физических узлах. Эмуляция на одной машине этого преимущества не даёт.

## Conda-pack

Скрипт `scripts/build_conda_env.sh` собирает упакованное Python-окружение для распространения на узлы кластера. На standalone-кластере с одним Worker'ом это не критично (Worker уже содержит PySpark), но команда показывает, как окружение упаковывается для отправки на распределённый кластер.
