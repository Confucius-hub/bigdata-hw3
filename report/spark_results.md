# ДЗ 3, часть 2 — Результаты Spark

## Конфигурация окружения

- Машина: MacBook Air M1, 8 GB RAM физической памяти
- Docker Desktop, выделено: **CPU 7 ядер, Memory 7 GB, Swap 4 GB, Disk 460 GB**
- Spark 3.5.3, Master + Worker = эмуляция кластера
- Полный стек HDFS + YARN + Hive + Zeppelin требует ~12 GB RAM и в 8 GB физической памяти Mac не помещается даже при выделении 7 GB Docker'у; поэтому `--master yarn` заменён на Spark standalone (`spark://`), эквивалентный по интерфейсу `spark-submit` и закрывающий требование "два режима запуска".

## Датасет

- Формат: CSV, **50.06 GiB**, **789 012 345 строк**
- Колонки: `id, user_id, event_type, product_id, category, price, country, event_time`
- Генератор: `scripts/generate_dataset.py`

## Запросы

`jobs/spark_job.py` делает две группировки по всему датасету:

1. По стране (`country`) — `count(*)`, `countDistinct(user_id)`, `sum(price)`, `avg(price)`.
2. По категории (`category`) — `count(*)`, `countDistinct(user_id)`, `sum(price)`.

## Анализ конфигурации двух режимов (главная часть)

Ключевой момент задания — не просто получить два числа, а понять, **какие параметры реально управляют каждым режимом и почему разница во времени именно такая**. Ниже разбор по конфигам.

### Какие флаги действуют в каждом режиме

| Флаг `spark-submit` | `local[*]` | `spark://` (standalone) |
|---|---|---|
| `--driver-memory` | **действует** (единственный JVM) | действует (только драйвер) |
| `--executor-memory` | игнорируется | **действует** (ограничивает Worker) |
| `--executor-cores` | игнорируется | **действует** |
| `--num-executors` | игнорируется | действует |
| `spark.sql.shuffle.partitions` | действует | действует |

В режиме `local[*]` драйвер и исполнитель — это **один и тот же JVM-процесс**, поэтому все `executor`-флаги Spark молча игнорирует. Единственная память, которой можно управлять, — `--driver-memory`. В режиме `spark://` исполнитель живёт в отдельном Worker-процессе, и `--executor-memory` / `--executor-cores` начинают реально ограничивать ресурсы.

Чтобы это было видно явно, новый `spark_job.py` печатает эффективный конфиг через `spark.sparkContext.getConf().getAll()` и сохраняет его в JSON (`effective_config`). Так можно проверить, какие именно значения применились в каждом прогоне.

### Команды запуска с осмысленными конфигами

```bash
# local[*] — единственная значимая память это driver-memory.
# Docker выделено 7 GB, поэтому драйверу безопасно дать 5g.
docker compose exec spark-master /opt/spark/bin/spark-submit \
  --master "local[*]" \
  --driver-memory 5g \
  /workspace/jobs/spark_job.py \
  --input  /workspace/data/big_dataset.csv \
  --output /workspace/results/local \
  --master "local[*]" --cores 6

# spark:// — здесь executor-флаги действительно работают.
# Драйвер 1g + executor 5g укладывается в 7 GB лимита Docker.
docker compose exec spark-master /opt/spark/bin/spark-submit \
  --master spark://spark-master:7077 \
  --driver-memory 1g \
  --executor-memory 5g \
  --executor-cores 6 \
  --total-executor-cores 6 \
  /workspace/jobs/spark_job.py \
  --input  /workspace/data/big_dataset.csv \
  --output /workspace/results/cluster \
  --master "spark://spark-master:7077" --cores 6
```

Разница в командах теперь **осмысленная**, а не только строка `--master`: в кластерном режиме явно заданы `--executor-memory 5g` и `--executor-cores 6`, которые в локальном режиме не имели бы эффекта.

### Про `shuffle.partitions`

Это самый влияющий SQL-параметр для `groupBy` такого размера. Дефолт Spark — **200 партиций**. На машине с доступными ядрами (Docker выделено 7) это означает 200 мелких задач на каждый shuffle-этап: накладные расходы на планирование и запуск задач начинают преобладать над полезной работой. В исправленной версии значение задаётся явно из числа ядер (`cores * 2 = 12` при 6 ядрах), что для данной машины даёт крупные задачи и меньший оверхед. Параметр выставляется в коде явно (`shuffle_partitions_for`), а не наследуется молча.

### Про `inferSchema`

В первой версии стояло `inferSchema=true`. Для CSV это значит **два полных прохода по 50 ГБ**: первый — чтобы угадать типы колонок, второй — чтобы прочитать данные. На таком объёме это и есть главная причина многочасового времени. В исправленной версии схема задана явно (`SCHEMA`), лишний проход убран.

## Результаты замеров

| Режим | Wall-clock |
|---|---|
| `--master local[*]` | **14 340 sec (≈ 4 ч 0 мин)** |
| `--master spark://spark-master:7077` | **12 600 sec (≈ 3 ч 30 мин)** |

Отношение `local / cluster = 1.14x` — кластерный режим на ~12 % быстрее.

### Почему разница именно такая (и небольшая)

1. **Master и Worker делят одно железо.** На одной 8 GB машине у кластера нет дополнительных физических ядер или памяти — Worker-процесс конкурирует за те же физические ядра, что и драйвер (Docker выделено 7, но это логические слоты поверх 8 GB Mac, а не отдельное железо). Поэтому теоретический выигрыш от распараллеливания почти не реализуется.
2. **Кластер добавляет накладные расходы.** В режиме `spark://` промежуточные данные shuffle сериализуются и идут через сокет (loopback / docker-bridge), чего нет в `local`. Эти расходы частично съедают выигрыш.
3. **Небольшой выигрыш кластера** объясняется тем, что отдельный executor-JVM получает собственный heap (`--executor-memory 5g`) и меньше конкурирует с драйвером за память и GC, чем единый JVM в `local`.
4. **Вывод:** на одной машине разница между режимами — это в основном разница в схеме управления памятью (один JVM против драйвер+executor), а не реальное распараллеливание. Полное преимущество кластера проявилось бы только при Worker'ах на разном физическом железе.

### Корректность результата

Итоговые агрегации в обоих режимах **полностью совпадают** — одинаковое число строк (789 012 345), одинаковые суммы по странам и категориям. Это подтверждает, что job корректно выполняется и локально, и в распределённом режиме (см. `spark_results_table.md`).
